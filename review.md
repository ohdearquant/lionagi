## Findings

1. MAJOR - The reaper can still block a legitimately refreshed/claimed play because the final write does not revalidate the current staleness/liveness inputs. In `reap_stale_plays()`, the candidate row is selected once by broad status membership at `lionagi/studio/services/lifecycle.py:364`, liveness is checked using that snapshot's `session_id` at `lionagi/studio/services/lifecycle.py:377`, and staleness is decided from that snapshot's `updated_at` at `lionagi/studio/services/lifecycle.py:390`. The guarded write then passes `expected_statuses=_REAPABLE_PLAY_STATUSES` at `lionagi/studio/services/lifecycle.py:420`, but `StateDB.update_status()` only checks that the current status is in the expected set (`lionagi/state/db.py:1505`) before writing the new status (`lionagi/state/db.py:1617`). That means a stale `prepared` row with `session_id=NULL` can be legitimately claimed after the scan, set to `running`, assigned a live child session, and refreshed to `updated_at=now`; the reaper will still write `blocked` because `running` remains in the expected set. I reproduced this against the actual `reap_stale_plays()` path: after simulating that concurrent claim immediately before `update_status()`, the reaper returned `1` and the final row was `status='blocked'` with the newly linked live session and a fresh `updated_at`. Fix by re-reading and revalidating current `status`, `session_id`, liveness, and `updated_at <= cutoff` immediately before the transition, and make the final status write conditional on that current stale predicate, not just status membership. The new CAS test only covers a concurrent terminal transition (`tests/apps_studio_server/test_lifecycle_play_reaper.py:308`); add a regression where a stale `prepared`/orphan row is claimed into a fresh live `running` row between scan and write and must not be blocked.

2. MINOR - `ended_at` is written before the CAS-protected status transition, so a skipped CAS can still mutate a row the reaper did not actually reap. `reap_stale_plays()` calls `db.update_play(play_id, ended_at=now)` at `lionagi/studio/services/lifecycle.py:404` before `db.update_status(... expected_statuses=...)` at `lionagi/studio/services/lifecycle.py:405`, while `update_play()` performs an unconditional `UPDATE plays SET ... WHERE id = :_id` at `lionagi/state/db.py:3051`. If another actor transitions the play to `merged` or another terminal state after the reaper's initial snapshot but before this write, `update_status()` correctly returns `False`, but the reaper can still overwrite `ended_at`/`updated_at` on the terminal row. Fix by moving `ended_at` into the same guarded transition/update path used for the stale-play write, or only setting it after the guarded transition has succeeded.
